<strong>Catalyst</strong> es un proyecto que busca que los esfuerzos
colaborativos florezcan en un todo mayor. Es una plataforma donde las personas
pueden colaborar en proyectos, compartir ideas y colaborar con otros.

<hr />

<h1 class="text-2xl">Tecnologías y Herramientas</h1>
<ul>
  <li>
    <h6 className="text-lg font-semibold">Next.js</h6>
    <p>
      Se eligió Next.js como framework porque sabíamos que gran parte del
      proyecto podría beneficiarse de la renderización del lado del servidor y
      de las acciones de servidor que ofrece App Router.
    </p>
  </li>
  <li>
    <h6 className="text-lg font-semibold">Docker</h6>
    <p>
      Se usó Docker para contenerizar la aplicación, garantizando la
      consistencia entre los entornos de desarrollo, staging y producción. Al
      empaquetar la aplicación y sus dependencias en un solo contenedor,
      redujimos los errores específicos del entorno y mejoramos la
      previsibilidad de la implementación. También facilitó la colaboración con
      otros desarrolladores, ya que preví que necesitaría ayuda a medida que el
      proyecto fuera más complejo.
    </p>
  </li>
  <li>
    <h6 className="text-lg font-semibold">Storybook</h6>
    <p>
      Para intentar mantener un alto nivel de calidad y consistencia, decidí
      usar Storybook para crear una biblioteca de interfaz de usuario para el
      proyecto. Esto me permitió crear un conjunto de componentes que se
      utilizarían en todo el proyecto y que serían estables, fáciles de usar y
      mantener.
    </p>
  </li>
  <li>
    <h6 className="text-lg font-semibold">Auth.js</h6>
    <p>
      Sabíamos que necesitaríamos implementar la autenticación y autorización de
      usuarios para el proyecto, así que decidí usar Auth.js, que ofrece una
      forma sencilla y segura de gestionar la autenticación y autorización de
      usuarios. En particular, usamos enlaces mágicos para permitir un inicio de
      sesión sin contraseña.
    </p>
  </li>
  <li>
    <h6 className="text-lg font-semibold">UploadThing</h6>
    <p>
      Subir imágenes es una tarea común en el proyecto. Para ello, implementamos
      UploadThing, una biblioteca de subida de imágenes ligera y rápida, que se
      adaptó perfectamente a nuestras necesidades.
    </p>
  </li>
  <li>
    <h6 className="text-lg font-semibold">
      React <span className="text-sm text-zinc-500 uppercase">Obsoleto</span>
    </h6>
    <p>
      Se eligió React como framework para el proyecto debido a su simplicidad,
      su adopción en toda la industria y su solidez. Este proyecto debía ser
      altamente interactivo, con numerosos cambios de estado continuos a medida
      que los usuarios interactuaban con el sitio, lo que hizo de React una
      excelente opción.
    </p>
    <p>
      En 2025, decidí adoptar un enfoque más moderno para el proyecto y opté por
      utilizar Next.js App Router.
    </p>
  </li>
  <li>
    <h6 className="text-lg font-semibold">
      Express
      <span className="text-sm text-zinc-500 uppercase">Obsoleto</span>
    </h6>
    <p>
      Express es el framework de referencia para crear API en Node.js. Es un
      framework ligero y rápido, fácil de usar y comprender. Dado que el backend
      de este proyecto podría volverse complejo, usar Express, que es poco
      convencional y permite mucha flexibilidad, fue una buena opción.
    </p>
    <p>
      En 2025, Express fue reemplazado por una combinación de acciones de
      servidor y rutas de API tras la implementación de Next.js App Router.
    </p>
  </li>
  <li>
    <h6 className="text-lg font-semibold">
      Nginx <span className="text-sm text-zinc-500 uppercase">Obsoleto</span>
    </h6>
    <p>
      Nginx se utilizó originalmente en este proyecto como proxy inverso y
      servidor de recursos estáticos. Ayudó a mejorar el rendimiento, gestionar
      la terminación HTTPS y gestionar el equilibrio de carga durante las
      primeras implementaciones. Esta configuración nos proporcionó mayor
      control sobre las estrategias de almacenamiento en caché y nos permitió
      servir la aplicación eficientemente en un entorno de producción.
    </p>
    <p>
      En 2025, dimos por obsoleto Nginx cuando el proyecto migró a una
      arquitectura sin servidor utilizando Next.js App Router, que gestionaba el
      enrutamiento y las funciones del servidor internamente.
    </p>
  </li>
</ul>

<h1 class="text-2xl">Roles y Responsabilidades</h1>
Este proyecto fue desarrollado por mí solo en 2023. En 2025, recibió una
revisión casi completa, con un enfoque en el rendimiento y la escalabilidad. Fui
responsable de cada faceta, desde el diseño hasta el desarrollo. Cabe destacar:
<ul>
  <li>
    <h6 className="text-lg font-semibold">Estructure de las vistas</h6>
    <p>
      Diseñé la forma en que se estructurarían los datos mostrados. Esta fue una
      tarea compleja, ya que el proyecto utiliza muchas capas de contenido,
      desde proyectos hasta las ramas y las publicaciones, por lo que era
      importante establecer una jerarquía y una estructura claras.
    </p>
  </li>
  <li>
    <h6 className="text-lg font-semibold">Comunicación front-back</h6>
    <p>
      Implementé los endpoint de la API del proyecto y me aseguré de que fueran
      eficientes y seguros. Al mismo tiempo, implementé métodos de query en el
      frontend que garantizaron que solo obtuviéramos los datos que
      necesitábamos. Utilicé Express, Passport y PostgreSQL, entre otras
      herramientas, para lograrlo.
    </p>
  </li>
</ul>

<h1 class="text-2xl">Desafíos y Soluciones</h1>
<ul>
  <li>
    <h6 className="text-lg font-semibold">Gestión del objetivo</h6>
    <p>
      Este fue un proyecto muy ambicioso y mantener un objetivo claro fue una
      tarea difícil. Tuve que decidir qué características priorizar y cuáles
      descartar. En muchas ocasiones, tuve que detenerme y reflexionar sobre si
      una determinada característica era útil para los objetivos del proyecto o
      si era solo una distracción. Por ejemplo, se descartó la idea de agregar
      ciertas "redes sociales", como los comentarios, pero se mantuvo un
      cronograma, ya que ayudaba con la idea de que las personas contribuyeran a
      los proyectos que les interesaban.
    </p>
    <p>
      Al final, creo que se conservó el objetivo inicial del proyecto como un
      sitio colaborativo, y el ciclo de desarrollo fue fluido y eficiente
      gracias a eso.
    </p>
  </li>
  <li>
    <h6 className="text-lg font-semibold">Dockerización</h6>
    <p>
      El proyecto tuvo muchos problemas al lidiar con la dockerización. Esta fue
      una de mis primeras experiencias al lidiar con contenedores en un proyecto
      full-stack, y tuve que solucionar y corregir muchos problemas. Después de
      mucho ensayo y error y lectura de documentación, pude contenerizar el
      proyecto.
    </p>
  </li>
  <li>
    <h6 className="text-lg font-semibold">Diseño de API</h6>
    <p>
      Diseñar la API fue una tarea compleja, ya que el proyecto tenía muchas
      partes interconectadas y era difícil saber cuánta información necesitaba
      cada parte, al mismo tiempo que se aseguraba que la API fuera segura y
      eficiente. En mi primer intento, fue un desastre, lo cual funcionó, pero
      no fue muy escalable ni eficiente.
    </p>
    <p>
      Reestructuré la API para que sea más flexible y escalable, con un enfoque
      en la claridad: cada punto final tiene un propósito claro, una entrada y
      una salida claras, y no hay contaminación de datos entre puntos finales.
      Esto permitió una API más eficiente y escalable, que también debería ser
      más fácil de mantener en el futuro.
    </p>
  </li>
  <li>
    <h6 className="text-lg font-semibold">Revisión de 2025</h6>
    <p>
      Para 2025, ya tenía mucha más experiencia en el campo y decidí revisar el
      proyecto. Sin embargo, pronto me di cuenta de que sería más rápido
      reiniciarlo por completo que intentar reescribir partes. Decidí adoptar un
      enfoque más moderno para el proyecto, centrándome en el rendimiento y la
      escalabilidad.
    </p>
    <p>
      Se eligió Next.js como framework. Implementé acciones de servidor y
      endpoints de API. Todo esto implicó una gran cantidad de "traducciones"
      entre la versión anterior y la nueva. A menudo, solo podía copiar la idea
      de la versión anterior y tenía que reescribir completamente la
      implementación.
    </p>
    <p>
      El resultado es un proyecto mucho más robusto y escalable, en el que, como
      desarrollador, puedo trabajar con mucha más eficiencia y confianza, y los
      usuarios tendrán una experiencia mucho mejor.
    </p>
  </li>
</ul>
