<strong>Catalyst</strong> es un proyecto destinado a permitir que los esfuerzos
colaborativos florezcan en un todo mayor. Es una plataforma donde las personas
pueden colaborar en proyectos y donde pueden compartir sus ideas y colaborar con
otros.

<hr />

<h1 class="text-2xl">Tecnologías y Herramientas</h1>
<ul>
  <li>
    <h6 className="text-lg font-semibold">React</h6>
    <p>
      Se eligió React como framework para el proyecto debido a su simplicidad,
      adopción en toda la industria y solidez. Este proyecto debía ser altamente
      interactivo, con muchos cambios de estado que sucedían continuamente a
      medida que los usuarios interactuaban con el sitio, lo que hizo que React
      fuera una gran opción para este proyecto.
    </p>
  </li>
  <li>
    <h6 className="text-lg font-semibold">Express</h6>
    <p>
      Express es el framework de referencia para crear API en Node.js. Es un
      framework liviano y rápido que es fácil de usar y comprender. Dado que el
      backend de este proyecto podría volverse complejo, el uso de Express, que
      es en gran parte sin opiniones y permite mucha flexibilidad, fue una buena
      opción.
    </p>
  </li>
</ul>

<hr />

<h1 class="text-2xl">Roles y Responsabilidades</h1>
Este proyecto lo desarrollé yo solo en 2023. Fui responsable de cada faceta del
mismo, desde el diseño hasta el desarrollo. Cabe destacar:
<ul>
  <li>
    <h6 className="text-lg font-semibold">Estructure de las vistas</h6>
    <p>
      Diseñé la forma en que se estructurarían los datos mostrados. Esta fue una
      tarea compleja, ya que el proyecto utiliza muchas capas de contenido,
      desde proyectos hasta las ramas y las publicaciones, por lo que era
      importante establecer una jerarquía y una estructura claras.
    </p>
  </li>
  <li>
    <h6 className="text-lg font-semibold">Comunicación front-back</h6>
    <p>
      Implementé los endpoint de la API del proyecto y me aseguré de que fueran
      eficientes y seguros. Al mismo tiempo, implementé métodos de query en el
      frontend que garantizaron que solo obtuviéramos los datos que
      necesitábamos. Utilicé Express, Passport y PostgreSQL, entre otras
      herramientas, para lograrlo.
    </p>
  </li>
</ul>

<hr />

<h1 class="text-2xl">Desafíos y Soluciones</h1>
<ul>
  <li>
    <h6 className="text-lg font-semibold">Gestión del objetivo</h6>
    <p>
      Este fue un proyecto muy ambicioso y mantener un objetivo claro fue una
      tarea difícil. Tuve que decidir qué características priorizar y cuáles
      descartar. En muchas ocasiones, tuve que detenerme y reflexionar sobre si
      una determinada característica era útil para los objetivos del proyecto o
      si era solo una distracción. Por ejemplo, se descartó la idea de agregar
      ciertas "redes sociales", como los comentarios, pero se mantuvo un
      cronograma, ya que ayudaba con la idea de que las personas contribuyeran a
      los proyectos que les interesaban.
    </p>
    <p>
      Al final, creo que se conservó el objetivo inicial del proyecto como un
      sitio colaborativo, y el ciclo de desarrollo fue fluido y eficiente
      gracias a eso.
    </p>
  </li>
  <li>
    <h6 className="text-lg font-semibold">Dockerización</h6>
    <p>
      El proyecto tuvo muchos problemas al lidiar con la dockerización. Esta fue
      una de mis primeras experiencias al lidiar con contenedores en un proyecto
      full-stack, y tuve que solucionar y corregir muchos problemas. Después de
      mucho ensayo y error y lectura de documentación, pude contenerizar el
      proyecto.
    </p>
  </li>
  <li>
    <h6 className="text-lg font-semibold">Diseño de API</h6>
    <p>
      Diseñar la API fue una tarea compleja, ya que el proyecto tenía muchas
      partes interconectadas y era difícil saber cuánta información necesitaba
      cada parte, al mismo tiempo que se aseguraba que la API fuera segura y
      eficiente. En mi primer intento, fue un desastre, lo cual funcionó, pero
      no fue muy escalable ni eficiente.
    </p>
    <p>
      Reestructuré la API para que sea más flexible y escalable, con un enfoque
      en la claridad: cada punto final tiene un propósito claro, una entrada y
      una salida claras, y no hay contaminación de datos entre puntos finales.
      Esto permitió una API más eficiente y escalable, que también debería ser
      más fácil de mantener en el futuro.
    </p>
  </li>
</ul>
