Welcome to another retrospective. These early months of 2026 have seen me focusing back on building projects, as I felt it had been too long since I had written any code. The result has been two new projects that reflect my growth as a developer, Archivist and Cook It.

### Archivist

Archivist is an alternate reality game set in the Cold War. Players stumble onto what appears to be a leak of classified CIA documents and, by following a trail of ciphers, hidden fragments and intercepted network traffic, piece together the fate of a missing operative and the handler who disappeared with her. It is the project I am most proud of, and also the one that pushed me the hardest.

The idea came from wanting to build something that was genuinely full-stack without feeling contrived about it. A lot of portfolio projects end up being full-stack in name only, with a backend that exists purely to justify calling it full-stack. With Archivist, I wanted every backend decision to earn its place in the player's experience. The answer validation API has rate limiting because brute-forcing answers would break the game, not because rate limiting is a box to tick. The Redis counters produce live visitor numbers that are woven into the lore of each document. The JWT cookie that tracks progress also hides a clue for players who think to inspect their own session data. Nothing is there just to be there.

What I didn't expect was how much of the challenge would be narrative rather than technical. Hiding a clue effectively is harder than it sounds. Too obvious and the puzzle collapses immediately; too obscure and the player gives up. Every piece of body text had to work as convincing Cold War bureaucratic writing and as a puzzle simultaneously. A statute reference reading "78-C" has to feel like administrative flavour until the moment it doesn't. Getting that balance right required more iteration than any of the technical work. I'm happy with where it landed, and I'm especially happy with how the story ends — but you'll have to find your way there yourself.

### Cook It!

Cook It! is a full-stack recipe platform built with Next.js and NestJS. Users can create and share recipes, browse and filter a public recipe list, follow other cooks, like and favorite recipes, and organize their favorites into collections. It is the most complex project I've built to date, and I'm very happy with how it turned out.

The backend is a NestJS REST API backed by PostgreSQL via Prisma, deployed to Fly.io. The frontend is a Next.js App Router application deployed to Vercel. One of the more interesting challenges was getting cookie-based authentication to work reliably across two different domains in production. Modern browsers are increasingly restrictive about cross-site cookies, and even with the correct `SameSite` and `Secure` settings, cookies set by the Fly.io backend were not reliably accepted by the Vercel frontend. The solution was to implement a Next.js rewrite proxy, routing all client-side API requests through the frontend domain so that cookies are always set on the same origin.

Another challenge was the distinction between server and client components in the App Router. A shared API utility was built with two variants — one for server components that forwards cookies from the incoming request, and one for client components that sends credentials automatically via the proxy. This kept the data fetching pattern consistent while respecting the server/client boundary.

On the feature side, the project gave me a lot of practice with patterns I had used before but never at this scale: cursor-based pagination, optimistic UI updates, infinite scroll, multi-step form wizards, and intercepting routes for modals. I also implemented Google OAuth alongside a full email/password flow with password reset via Nodemailer, which was my first time building a complete authentication system from scratch without relying on a library like Auth.js.

Overall, Cook It! gave me a much deeper appreciation for what goes into building and deploying a full-stack product end to end, and I'm excited to keep building on top of it.

### Code Reviews

After the work on Cook It! and Archivist, I updated many of my older projects, such as ESIA, Oura or Catalyst. Some of the changes were minor stylistic tweaks, but there were also bug fixes, performance improvements and new features. For example, ESIA had a lot of redundant error handling code that was removed, and Catalyst had certain bugs in its fetching logic fixed.

### Portfolio Projects Refresh

Lastly, the portfolio projects section has seen a refresh. Some of my older projects have been removed, and new projects have been added. This felt like a logical progression that shows my current skills and all the knowledge I've gathered over the years. The portfolio page also saw some small stylistic tweaks; the color scheme has been updated to be more subdued, as I felt the vibrant colors didn't fit a portfolio website. Similarly, some hover effects have been tweaked to make them more subtle. Overall, I'm happy with the changes, and the site feels more polished and professional.
